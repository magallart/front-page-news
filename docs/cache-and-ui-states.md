# Snapshot, Cache, and UI State Strategy

## Purpose

This document describes the current data-delivery strategy for news and sources:

- server-side persisted snapshots in Vercel Blob
- client-side persisted snapshots in IndexedDB
- in-memory request caches
- stale-while-revalidate behavior in the Angular UI

It is the operational reference for freshness rules, fallback behavior, and visible UI states.

## Architecture Overview

The app resolves news through four layers, in this order:

1. in-memory cache in `NewsService`
2. persisted client cache in `IndexedDbSnapshotCache`
3. persisted remote snapshot in Vercel Blob
4. fresh network request to `/api/news`

The current sources flow remains simpler:

1. in-memory cache in `SourcesService`
2. `/api/sources`

Base snapshots for `home`, navbar sections, and `sources` are regenerated by a protected Vercel cron.

## Canonical Snapshot Keys

Snapshot keys are shared across client, API, and Blob storage.

News format:

```text
news:id=<id>:section=<section>:source=<sorted-source-ids>:q=<query>:page=<page>:limit=<limit>
```

Sources format:

```text
sources:default
```

Normalization rules:

- empty values become `-`
- `section` and `q` are normalized to lowercase
- `sourceIds` are deduplicated, lowercased, sorted alphabetically
- `id` preserves original casing

## Freshness Policy

### Remote snapshots in Blob

- `staleAt = generatedAt + 15 minutes`
- `expiresAt = generatedAt + 36 hours`

### Client persisted snapshots in IndexedDB

- `staleAt = generatedAt + 15 minutes`
- `expiresAt = min(snapshot.expiresAt, generatedAt + 12 hours)`

### In-memory cache

- `NewsService`: `60_000 ms`
- `SourcesService`: `300_000 ms`

## News Resolution Flow

### `NewsService`

For `getNews(query, { forceRefresh })`:

1. If `forceRefresh=true`, skip hydration caches and request fresh data.
2. Reuse in-memory result when memory TTL is valid.
3. If the in-memory entry is stale, emit it immediately and revalidate in background.
4. If there is no usable memory entry, try IndexedDB.
5. If IndexedDB misses, try remote Blob snapshot.
6. If a hydrated snapshot is fresh, render it and stop.
7. If a hydrated snapshot is stale, render it and request fresh data.
8. If there is no renderable cached data, wait for fresh network data.

### Degradation rules

- IndexedDB read failures are ignored and hydration continues with the next layer.
- IndexedDB write failures are ignored and visible data is still delivered.
- Remote snapshot fetch or parse failures are ignored and the flow falls back to network.
- Fresh network failure only becomes user-visible when no renderable cached data exists.

This is intentional: local persistence is an optimization, not a hard dependency for rendering.

## Server Snapshot Behavior

### `/api/news`

- Reads a persisted snapshot first when available and not expired.
- Falls back to live feed aggregation when the snapshot is missing or expired.
- Keeps a per-query in-memory cache and in-flight dedupe in the server runtime.
- Supports forced refresh with `x-news-force-refresh: 1`.

### `/api/sources`

- Reads a persisted sources snapshot first when available and not expired.
- Falls back to catalog-derived live response otherwise.

### Cron regeneration

`GET /api/cron/regenerate-snapshots`

- requires `Authorization: Bearer <CRON_SECRET>`
- regenerates:
  - home snapshot
  - all base section snapshots from `BASE_SNAPSHOT_SECTION_SLUGS`
  - `sources:default`

The regeneration process does not persist degraded news snapshots when:

- the payload is empty
- warnings contain blocking feed failures such as timeout, fetch failure, or parse failure

## UI State Model

### Store-level freshness signals

`NewsStore` exposes:

- `isHydrated`
- `isInitialLoading`
- `isRefreshing`
- `isShowingStaleData`
- `hasFreshUpdateAvailable`
- `lastUpdated`

### Page-level view states

`Home` and `Section` still resolve their main visual state from:

- `loading`
- `error`
- `warnings`
- item count

Supported states:

- `loading`
- `empty`
- `error_total`
- `error_partial`
- `ready`

### Skeleton rule

Skeletons are allowed only when there is no renderable data from any layer.

If any cached or snapshot-backed data exists, the page renders content immediately and uses refresh indicators instead of clearing the UI.

### Refresh indicators

The reusable `app-news-refresh-indicator` exposes two non-blocking signals:

- background refresh status when stale content is visible and revalidation is running
- dismissible fresh-update banner when a background refresh replaces visible content with newer data

## Manual Refresh and Invalidation

Recommended usage:

- use `NewsStore.refresh(query)` for user-driven refreshes
- use `SourcesStore.refresh()` for sources reloads
- use `forceRefresh` only when freshness must bypass hydration layers
- use `invalidateBySection(sectionSlug)` for scoped in-memory invalidation
- use `clear()` only for broad invalidation events

## Environment and Deployment Notes

Relevant variables:

- `SNAPSHOT_BLOB_BASE_URL`
  - public read base URL used by server and browser snapshot readers
- `BLOB_READ_WRITE_TOKEN`
  - Vercel Blob write token used by cron regeneration
- `CRON_SECRET`
  - bearer secret required by `/api/cron/regenerate-snapshots`
- `NEWS_PERF_LOGS`
  - optional server flag to print cache/perf diagnostics in `/api/news`

The Angular environment value `snapshotBlobBaseUrl` must resolve to the same public Blob base URL used by the server-side reader.

## Maintenance Notes

- Keep TTL values centralized in constants.
- Preserve canonical snapshot-key generation across all layers.
- Treat Blob and IndexedDB as optional resilience layers, not rendering prerequisites.
- Update this document when freshness thresholds, fallback rules, or visible refresh cues change.
